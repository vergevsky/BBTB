# RulesEngine Baseline Resources

Этот каталог содержит pre-compiled signed baseline rules, вшитые в `.app` bundle
через `Bundle.module`. После Phase 8 W6 файлы — **real signed artifacts**
(не placeholders): `RulesEngineCoordinator.bootstrap()` копирует их в App Group
cache на first-launch, sing-box engine читает SRS-binaries и блокирует
domains/IPs **до первого server fetch** без сетевой зависимости.

Pitfall 6 Option A: build artifacts committed в repo. CI/`tuist generate` НЕ
зависит от sing-box CLI — developer перегенерирует локально когда меняется
`baseline-rules.json`.

## Files

| File | Origin | When updated |
|------|--------|--------------|
| `baseline-rules.json` | Hand-edited source of truth (Phase 8 W6) | Когда админ курирует block list (раз в релиз / по hotfix) |
| `baseline-rules-manifest.json` | Generated by `build-baseline-rules.sh` | Auto on script run |
| `baseline-rules-manifest.json.sig` | Generated by `build-baseline-rules.sh` (Ed25519 detached, 64 bytes) | Auto on script run |
| `bbtb-baseline-{block,never,always}.srs` | Generated by `build-baseline-rules.sh` (sing-box rule-set compile) | Auto on script run |
| `bbtb-baseline-{block,never,always}.srs.sig` | Generated by `build-baseline-rules.sh` (Ed25519 detached, 64 bytes) | Auto on script run |

## Regenerate workflow (Phase 8 W6)

### Production signing mode

```bash
# 1. Установить toolchain (один раз):
brew install sing-box        # CLI v1.13.x — SRS compile
brew install openssl@3       # 3.x с -rawin Ed25519 support (LibreSSL не подходит)
brew install jq

# 2. Указать production private key (хранится вне repo: 1Password / Keychain).
export BBTB_BASELINE_SIGNING_KEY=/secure/path/to/bbtb-rules-private.pem

# 3. Запустить build:
bash BBTB/scripts/build-baseline-rules.sh

# 4. Commit обновлённые файлы:
git add BBTB/Packages/RulesEngine/Sources/RulesEngine/Resources/
git commit -m "chore(08): rebuild baseline rules"
```

В production mode `PublicKey.swift` **не изменяется** — assumed что в нём уже
зашита pubkey соответствующая production private key.

### Ephemeral mode (developer convenience / CI sanity)

Если `BBTB_BASELINE_SIGNING_KEY` не set, script генерирует **ephemeral**
Ed25519 keypair в `$TMPDIR`, использует его для signing AND обновляет
`publicKeyBytes` в `PublicKey.swift` соответствующей derived pubkey:

```bash
bash BBTB/scripts/build-baseline-rules.sh
# WARNING: BBTB_BASELINE_SIGNING_KEY not set — generated ephemeral keypair.
# ...
# Updating PublicKey.swift with ephemeral pubkey bytes…
# PublicKey.swift updated: placeholder sequence absent.
```

Ephemeral mode — для autonomous CI runs / worktree environments / Phase 8 W6
self-contained execution. Для production deployment перейти на production mode.

## NOT for CI

Build script намеренно НЕ wired в CI (Pitfall 6 Option A). CI запускается
против committed artifacts. Developer regenerates локально когда
`baseline-rules.json` меняется.

## W7 invariants (`validate-r1-r6.sh`)

Phase 8 W7 расширит `validate-r1-r6.sh` следующими R12 hard-gates:
- `.sig` файлы НЕ должны быть 64 zero bytes (placeholder detection).
- `PublicKey.swift publicKeyBytes` НЕ должен содержать sequential pattern
  `0x00, 0x01, 0x02, 0x03` (W1 placeholder detection).

После запуска `build-baseline-rules.sh` (любой mode) оба guard'а PASS.
