# GeoIP-детект (Этап 1)

**Summary**: Этап 1 методики — серверная проверка геолокации по IP. Это **единственный сигнал, без которого** методика не даёт жёсткий вердикт «обход выявлен». Главный фронт защиты для нашего проекта — стабильное GeoIP-соответствие.

**Sources**: ocr_methodika_vpn_proxy.md (раздел 5)

**Last updated**: 2026-05-11

---

## Почему это главное

Из матрицы решений (см. [[rkn-methodology-document]]):

> Если GeoIP не выявил аномалию — никакая комбинация прямых и косвенных признаков **сама по себе** не приводит к вердикту «обход выявлен».

То есть если **сервер видит** клиента с российским IP без признаков hosting/proxy — даже наличие `utun*`-интерфейса (см. [[apple-detection-surface]]) не превращается в жёсткий вердикт. В худшем случае — «требуется доп. проверка».

## Источники GeoIP по методике

| Источник | Статус |
|----------|--------|
| **РАНР** — «Реестр адресно-номерных ресурсов сети Интернет» | Главный (после ввода в эксплуатацию) |
| **MaxMind** | Альтернативная коммерческая БД (до РАНР) |
| **IP2Location** | Альтернативная коммерческая БД |
| Дополнительные коммерческие/внутренние источники | Опционально для повышения точности |

Репо [[rkn-detection-methodology|xtclovver/RKNHardering]] использует `api.ipapi.is` и `iplocate.io` — это конкретные API того же класса.

## Какие сигналы анализируются

Из раздела 5.4:

| Сигнал | Что означает |
|--------|--------------|
| Страна и регион по IP | Соответствие заявленному местоположению пользователя |
| Аномалии: частая смена стран, резкие изменения локаций | VPN-hopping |
| **ASN и тип сети** (Autonomous System Number — номер автономной системы) | Сравнение с диапазонами дата-центров и хостинг-провайдеров |
| Репутационные списки VPN/Proxy/TOR exit nodes | Прямое попадание = детект |
| Белые списки корпоративных сетей и CDN | Снижение ложных срабатываний |

## Решающая логика (раздел 5.4)

| Серверный GeoIP | Клиентский self-report | Решение |
|-----------------|------------------------|---------|
| Устройство в РФ | Устройство в РФ | **VPN не выявлен** |
| Зарубежом | В РФ | **VPN выявлен** |
| Зарубежом | Зарубежом | Требуется доп. проверка |
| IP в репутационном списке / hosting | Любой | **VPN выявлен** независимо |

Клиентский self-report — это не наш контроль (это GPS или мобильная сеть пользователя). Сервер сравнивает «откуда я тебя вижу» с «где ты себя считаешь».

## Угрозы для нашего проекта

### 1. Hosting/datacenter IP exit-серверов

Если наши VPN-серверы стоят на стандартных хостинг-провайдерах (Hetzner, DigitalOcean, Vultr, OVH) — их IP-диапазоны **известны** GeoIP-провайдерам как «hosting». Это **прямое попадание** в условие из раздела 5.4 (последняя строка таблицы) и даёт **жёсткий «VPN выявлен»** независимо от страны.

**Это для нашего проекта — главная архитектурная угроза.**

### 2. Репутационные списки VPN-провайдеров

Известные VPN-сервисы (NordVPN, Surfshark, ExpressVPN) — все их IP в репутационных списках. Если мы покупаем уже использовавшийся IP — попадаем туда же.

### 3. ASN дата-центров

Даже если IP не помечен как «hosting», его ASN (через BGP) укажет на дата-центр. Это slower-moving сигнал, но реалистичный.

### 4. Российская страна в IP важна

Из таблицы решений: «По результатам серверной и клиентской проверки устройство находится зарубежом» → «Требуется дополнительная проверка». То есть просто **«не РФ»** ещё не «выявлен» — но переводит в очередь дополнительных проверок (где сработают прямые/косвенные признаки и тогда уже будет «выявлен»).

## Что это значит для архитектуры

1. **Resident IP > Datacenter IP**. Лучше домашние/мобильные IP-адреса, чем стандартный хостинг. Это дороже, но критично против GeoIP-детекта. Источник этого варианта — резидентные прокси-провайдеры (см. ниже).
2. **Российский exit для российских пользователей?** — кажется бредом, но методика это явно описывает как **отсутствие сигнала**. Если exit-сервер тоже в РФ (например, обходит блокировку через российскую же AS, но не транзит ТСПУ), GeoIP не даст аномалии. Это спорный вариант, может конфликтовать с целью обхода [[tspu]].
3. **Регулярная ротация IP** — не помогает, если новые IP тоже из тех же hosting-диапазонов.
4. **CDN-фронтинг** (см. [[anti-dpi-techniques]]) частично помогает: сервер приложения видит IP CDN, а не VPN-сервера. Cloudflare/Fastly попадают в whitelist «легитимных CDN».

## Резидентные прокси-провайдеры (вне MVP)

Методика прямо признаёт пробел (раздел 6.6): «Резидентные прокси и сценарии, где внешний IP выглядит как обычный адрес провайдера». Это коммерческий сервис — IP-адреса реальных пользователей сдаются как exit-точки. Цена высокая, юр-вопросы серьёзные, **на MVP не делаем**.

## Альтернативные источники локации (раздел 5.5)

Методика разрешает использовать:
- Идентификаторы базовых станций мобильной сети (PLMN)
- BSSID Wi-Fi точек доступа
- GPS-координаты (только с согласия пользователя)

Эти данные **дополняют** GeoIP, не заменяют. Полезно знать: если приложение получило GPS-разрешение — оно может перепроверить серверный GeoIP против реальных координат пользователя. То есть **GPS-разрешение в сторонних приложениях усиливает детект**.

## Ограничения GeoIP, явно перечисленные в методике (раздел 5.6)

- Ограниченная точность для мобильных сетей, CGNAT, корпоративных сетей, пограничных регионов
- Корпоративные легальные VPN определяются как обход блокировок (false positive)
- CDN и глобальные сервисы могут искажать местоположение
- **Резидентные прокси не выявляются по ASN и hosting**
- Новые VPN-серверы появляются быстрее, чем обновляются репутационные базы
- Роуминг: точка выхода в стране пребывания

## Что планируем

Серверная архитектура — вне scope основного промта, но ключевые принципы:
- Не использовать публичных hosting-провайдеров для exit-серверов
- Не использовать IP, уже засвеченные другими VPN-сервисами
- Рассматривать resident-IP как стратегию post-MVP
- Мониторить попадание наших IP в репутационные списки

Конкретный план серверной инфраструктуры — отдельный документ, не охвачен текущим первичным ингестом.
