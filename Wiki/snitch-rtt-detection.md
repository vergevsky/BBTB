# SNITCH (RTT-триангуляция)

**Summary**: Дополнительный метод детекта VPN из раздела 10 методики РКН. Идея — измерять RTT (Round-Trip Time) от сервера до клиента и сравнивать с задержками до «контрольных точек». Аномально высокий RTT для заявленной геолокации = признак VPN. Метод ОС-независимый, работает на любых устройствах включая iOS/macOS.

**Sources**: ocr_methodika_vpn_proxy.md (раздел 10.1), https://github.com/xtclovver/RKNHardering (имплементация SNITCH β)

**Last updated**: 2026-05-11

---

## Что это

SNITCH — Server-side Non-intrusive Identification of Tunnelled Characteristics. Если расшифровать:

- **Server-side** — детект происходит на стороне сервера приложения (не на клиенте)
- **Non-intrusive** — не требует особых прав на клиенте, не использует системные API
- **Identification** — задача — определить факт туннелирования
- **Tunnelled Characteristics** — характеристики, появляющиеся при туннелировании трафика

## Как работает

Базовая идея:

1. Сервер измеряет RTT (время отклика) между собой и клиентом
2. Сервер знает «геолокацию» клиента по IP (через [[geoip-detection|GeoIP]])
3. Сервер измеряет RTT до набора «контрольных точек» (`landmarks`) с известными координатами
4. Триангуляция: задержки до клиента должны быть согласованы с задержками до landmark'ов, близких к заявленной геолокации
5. Если RTT до клиента **аномально высокий** для заявленной геолокации → клиент трафик гонит через дополнительный hop (VPN) → детект

## Имплементация в RKNHardering (β)

Репо [[rkn-detection-methodology|xtclovver/RKNHardering]] реализует упрощённую клиентскую версию (отключена по умолчанию):

Цели для пинга:
- **РФ**: `yandex.ru`, `mail.ru`, `vk.com`, `sberbank.ru`, `gosuslugi.ru`
- **Зарубежные**: `facebook.com`, `github.com`, `twitter.com`, `reddit.com`, `instagram.com`

Логика:
- Медианный RTT до RU-хостов **> 80 мс** → устройство, вероятно, не в РФ
- Высокий jitter (> 60 мс) снижает уверенность вывода
- Результат переводит вердикт в `NEEDS_REVIEW`, но сам по себе не даёт `DETECTED`

Это **клиентская** реализация (наоборот к серверной из методички), но идея та же — измерять задержки и триангулировать.

## Применимость к нашему проекту

### Платформенно-независимый

SNITCH работает по сети, **не зависит от ОС** клиента. iOS sandbox его не блокирует — измерения делает сервер на основе сетевой стороны соединения.

### Что это значит для архитектуры VPN-серверов

Если VPN-сервер находится **далеко** от заявленной геолокации клиента (например, exit в Нидерландах для пользователя в РФ), RTT от приложения до своего бэкенда увеличивается на 2× routing-time:

```
App в РФ → VPN-сервер NL → Сервер банка в РФ → ответ → VPN-сервер NL → App
```

Это может добавить 30-100 мс. Если landmark'и для РФ географически близки (~5-30 мс типичный RTT внутри РФ), задержка через западный exit будет аномальной.

### Контрмеры

| Контрмера | Эффективность |
|-----------|---------------|
| **Exit-сервер близко к ожидаемой геолокации** | ✅ высокая |
| **Множественные exit-серверы в разных регионах** | ⚠ ограниченная |
| Маскировка SNI (не помогает RTT-измерениям) | ❌ |
| uTLS, padding, фрагментация | ❌ |

То есть SNITCH **не обходится** нашими [[anti-dpi-techniques|anti-DPI техниками]] (это другой класс защиты — анти-DPI про маскировку трафика, SNITCH про физику задержек).

## Текущая реальность

Полноценная серверная реализация SNITCH требует:
- Инфраструктуры landmark'ов с точно известными координатами
- Регулярных измерений
- ML/статистики для интерпретации

В РФ на момент написания методички (см. документ) — **внедрено фрагментарно** или только тестируется. Это отдалённая угроза, но реальная.

## Что планируем

1. Признать SNITCH как **архитектурную угрозу** на горизонте
2. Учесть в roadmap: при выборе exit-серверов учитывать гео-близость к пользователю
3. На MVP игнорируем — не вкладываемся специально
4. В v1.1+ (smart auto-select, см. [[release-roadmap]]) можно добавить метрику «гео-близость exit'а к пользователю» — это попутно снижает SNITCH-сигнал

## Что НЕ помогает против SNITCH

- ❌ Любая маскировка трафика на уровне packet'ов
- ❌ Смена протокола (VLESS, WireGuard, Hysteria2)
- ❌ Смена транспорта (XHTTP, gRPC, WebSocket)
- ❌ CDN-фронтинг (CDN добавляет ещё один hop — может даже **усилить** RTT-сигнал)

SNITCH — это физика. Свет идёт со скоростью света, RTT — приближённая функция расстояния.

## Что помогает

- ✅ Физическое расстояние от exit-сервера до клиента минимизировано
- ✅ Прямой пиринг с российскими IX (Internet Exchange) — снижает routing-overhead
- ✅ В перспективе — резидентные exit-IP в той же стране, что и клиент

## HTTP-заголовки (раздел 10.2 методики)

Параллельный дополнительный метод — анализ заголовков `X-Forwarded-For`, `Forwarded`, `Via`. Применяется с осторожностью, потому что CDN их легитимно формируют. Не угрожает нам напрямую — наш трафик идёт зашифрованным внутри туннеля, и сервер приложения видит запрос как от exit-сервера, без proxy-заголовков.
