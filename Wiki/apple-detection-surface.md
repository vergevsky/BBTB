# Поверхность детекта на iOS и macOS

**Summary**: Все методы детектирования VPN/Proxy из методики РКН, применимые к iOS и macOS. Каждый сигнал — конкретный API, оценка реальной опасности и проектное решение. Это таблица угроз нашему конкретному приложению.

**Sources**: ocr_methodika_vpn_proxy.md (разделы 6.5, 7.5, 7.6, 7.7, 8.4), https://github.com/xtclovver/RKNHardering

**Last updated**: 2026-05-11

---

## Сводка

| Платформа | Прямой детект | Косвенный детект | Маршрутизация | Native |
|-----------|---------------|-------------------|---------------|--------|
| **iOS** | очень ограничен | через `__SCOPED__`, `NWPathMonitor` | **не применим** | нет |
| **macOS** | через `getifaddrs()` + проверки | развитее iOS | применим полностью | через `enforceRoutes` |

iOS защищён архитектурно sandbox'ом. macOS уязвимее — там доступны API, эквивалентные Android.

## iOS — прямые признаки (раздел 6.5 методики)

### `CFNetworkCopySystemProxySettings()`

Возвращает словарь с текущими настройками системного proxy.

| Что детектируется | Логика |
|-------------------|--------|
| IP и port proxy в настройках | Весь трафик идёт через proxy → детект |

**Применимость к нам**: В нашей архитектуре трафик ходит через [[architecture|NEPacketTunnelProvider]] (Layer 3 туннелирование), а не через системный proxy. То есть наш VPN **не выставляет** системный proxy в настройках iOS. Этот сигнал не должен срабатывать на наших пользователях.

**Контрмера**: убедиться, что `libbox.xcframework` не выставляет системный proxy через `kCFNetworkProxiesHTTPEnable` или аналогичные ключи. Проверить эмпирически на dev-устройстве.

### Прочее

Методика констатирует: «На iOS доступ к системным данным существенно ограничен». Других прямых признаков методичка не упоминает. Это **сильная сторона iOS** — sandbox-модель работает в нашу пользу.

## iOS — косвенные признаки (раздел 7.5)

### Ключ `__SCOPED__` в `CFNetworkCopySystemProxySettings()`

Внутри возвращаемого словаря содержатся имена активных интерфейсов.

| Имя в `__SCOPED__` | Сигнал |
|--------------------|--------|
| `utun*` | VPN |
| `tap*` | VPN |
| `tun*` | VPN |
| `ppp*` | VPN |
| `ipsec*` | VPN |

**Применимость к нам**: При активном PacketTunnelProvider у нас будет интерфейс `utun*`. Это **обнаружимый сигнал**. Скрыть его без jailbreak на iOS невозможно.

**Контрмера**: невозможна на iOS. Полагаемся на матрицу: пока GeoIP показывает РФ, один косвенный сигнал не даёт «обход выявлен» (см. [[rkn-methodology-document]]).

### Параметр `P2P` на интерфейсе

Косвенный признак VPN. Зависит от конкретной конфигурации туннеля.

**Решение (2026-05-11)**: при настройке `NEPacketTunnelNetworkSettings` проверить и **не выставлять** `P2P=true`. Работа на 30 минут в v0.1, закрывает один косвенный сигнал. См. [[security-gaps]] R6.

### `NWPathMonitor`

```swift
let monitor = NWPathMonitor()
monitor.pathUpdateHandler = { path in
    // Анализ path.availableInterfaces на:
    // - .other type
    // - имя содержит "utun"
}
```

Любое приложение может подписаться на `NWPathMonitor` и видеть наш `utun`-интерфейс среди `path.availableInterfaces` с типом `.other`.

**Контрмера**: невозможна на iOS. То же, что для `__SCOPED__`.

### `NEVPNManager`

```swift
NEVPNManager.shared().loadFromPreferences { ... }
```

Возвращает информацию **только о VPN, сконфигурированных через само приложение** или доступных по модели `NetworkExtension`. Требует специальных entitlements.

**Применимость к нам**: внешние приложения **не увидят** наш VPN через `NEVPNManager` — он сконфигурирован через **наше** приложение и приватен для других. Это не угроза.

### iCloud Private Relay

Apple-сервис, позволяющий обходить блокировки. Методичка прямо говорит: «не должен автоматически классифицироваться как запрещенный VPN, хотя его использование и позволяет обходить блокировки. Работа с этим сервисом требует отдельной логики обработки для снижения рисков ложных срабатываний».

**Применимость к нам**: edge case. Если у пользователя одновременно включены iCloud Private Relay и наш VPN — поведение детекторов непредсказуемо. Open question в [[security-gaps]].

## iOS — маршрутизация (раздел 7.6)

Методика прямо констатирует:

> Анализ таблиц маршрутизации не применим для iOS, поскольку доступ приложений к информации о других приложениях и системных настройках ограничен.

**Применимость к нам**: ✅ **iOS защищён**. Никакое приложение в обычном sandbox не может прочитать таблицу маршрутизации устройства. Угрозы нет.

## iOS — DNS (раздел 7.7)

Методика называет аномалии DNS «уточняющим признаком, но сам по себе недостаточен». Признаки:

- DNS-серверам назначаются локальные адреса (loopback)
- Все DNS-запросы направляются в виртуальный интерфейс

**Применимость к нам**: при активной [[dns-strategy|DoH внутри туннеля]] system DNS на iOS может выглядеть «изменённым». Это слабый сигнал и **не** даёт прямого детекта без других признаков.

Методика не выделяет отдельный раздел «macOS прямые признаки» — macOS попадает в этап 3 (раздел 8.4). Прямые признаки macOS пересекаются с косвенными.

## macOS — раздел 8.4 (Этап 3 «Широта охвата»)

### `getifaddrs()`

Стандартный POSIX API. Позволяет перечислить все сетевые интерфейсы, включая виртуальные.

**Применимость к нам**: при активном PacketTunnelProvider будет виден интерфейс типа `utun*`. **Обнаружимо**, скрыть без root невозможно.

### Анализ таблицы маршрутизации

В отличие от iOS, на macOS таблица маршрутизации **доступна** обычным приложениям (через `getfsstat`, `sysctl(CTL_NET, PF_ROUTE, ...)`, или просто `route -n get` через NSTask).

| Аномалия | Что означает |
|----------|--------------|
| Default route через нестандартный интерфейс | VPN active |
| Несколько default routes | Split tunneling |
| Маршруты через виртуальные интерфейсы | VPN active |
| Нестандартные MTU (например, 1350) | Туннелирование |

**Применимость к нам**: при активном PacketTunnelProvider — все эти признаки сработают. Это **сильный сигнал**, но он не превышает «косвенный» в матрице.

### Transparent Proxy API

Отдельный тип сетевого контура (`NEAppProxyProvider` — наша [[architecture|архитектура]] использует его как опциональный для macOS split-tunneling). Сам факт активного TransparentProxy — сигнал.

**Применимость к нам**: только если пользователь включил `AppProxyProvider`. По умолчанию выключено.

### `Network Extensions`

Факт наличия Network Extensions, разрешённых пользователем, — технический признак.

**Применимость к нам**: ✅ при установке нашего приложения и согласии пользователя на VPN-разрешение это **видно** другим приложениям через специальные API на macOS.

**Контрмера**: невозможна — это часть архитектуры платформы.

### `enforceRoutes` — принятый trade-off

Цитата методики:

> Принудительная маршрутизация `enforceRoutes` может рассматриваться как дополнительный технический признак.

В нашей архитектуре [[kill-switch]] использует `NEVPNProtocol.enforceRoutes = true` как обязательную защиту от split DNS-leak.

**Решение (2026-05-11)**: оставляем `enforceRoutes = true` по дефолту. Защита от DNS-leak приоритетнее, snижение детектируемости — через GeoIP-маскировку, не через ручное отключение защит. См. [[security-gaps]] R4.

**Для tech-savvy пользователей**: с v0.10 в [[ux-specification|Расширенных]] появится опциональный тоггл «Отключить принудительную маршрутизацию» (см. [[security-gaps]] R5).

## Главный вывод

На iOS мы детектируемы по `utun*`-интерфейсу, но **скрыть это невозможно** без jailbreak. На macOS поверхность шире (маршруты, `enforceRoutes`, `getifaddrs`). В обоих случаях защита держится на **матрице решений**: пока GeoIP показывает РФ, один косвенный сигнал не превращает «обход не выявлен» в «обход выявлен».

Поэтому стратегический приоритет — **GeoIP-маскировка**. См. [[geoip-detection]].
