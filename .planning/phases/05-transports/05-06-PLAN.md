<objective>
Wave 6 — архитектурный refactor для устранения cyclic dependency перед Wave 7 integration.

**Проблема**: D-14/D-15 требуют `PoolBuilder` (в ConfigParser) вызывать
`VLESSTLS.ConfigBuilder.buildOutbound(from: parsedXxx, transport: ..., tag: ...)`. Это требует
ConfigParser → import Protocols/*. Но протокольные пакеты должны принимать `ParsedXxx` структуры
(определены в ConfigParser сегодня) — Protocols/* → import ConfigParser. Получается cycle:
ConfigParser ↔ Protocols.

**Решение**: Переносим `ParsedXxx` structs + `AnyParsedConfig` enum + `UnsupportedReason` enum
+ `ImportedServer` enum **из ConfigParser в VPNCore**. После переноса:
- VPNCore содержит data model (типы Parsed*, AnyParsedConfig, TransportConfig)
- ConfigParser → VPNCore (для возврата AnyParsedConfig из парсеров)
- Protocols/* → VPNCore (для accept ParsedXxx в buildOutbound — Wave 7)
- ConfigParser → Protocols/* (для вызова buildOutbound из PoolBuilder — Wave 7)

Cycle устранён: ConfigParser → Protocols → VPNCore (один направление DAG).

Wave 6 — **pure refactor**: ZERO behavior changes. Перемещение типов между файлами + import
substitution в downstream callers. Все Phase 4 + Wave 0-5 тесты должны PASS байт-в-байт после
этого refactor. Если хотя бы один тест ломается — refactor неправилен.

Output: 1 новый файл в VPNCore (ParsedConfigs.swift) + 1 тест-файл + 7 модифицированных
файлов в ConfigParser (URI parsers + PoolBuilder + ImportedServer.swift — последний удаляется
или становится пустым shell-ом).
</objective>
