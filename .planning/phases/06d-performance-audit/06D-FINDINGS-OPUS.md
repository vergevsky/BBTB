# Phase 6d Audit â€” OPUS Pass

## 1. Executive summary

The codebase is well-engineered around the Phase 6c invariants, but five repeated patterns dominate the user-perceived "heaviness" since Phase 5:

- **Connect tap latency is dominated by a 1-second polling loop in `TunnelController.connect()`** â€” every successful connect waits at least one full second after `.connected` arrives, and we run up to 30 of these sleeps even when iOS has already fired the `NEVPNStatusDidChange` notification within ~200 ms. This is the single largest contributor to the "tap is slow" feeling.
- **Cold start does six-to-eight XPC trips before the UI becomes interactive** (`MainScreenViewModel.init` seed + `OnDemandMigrationTask.runIfNeeded` + `SwiftDataFailoverProvider` task + `TunnelController.startReachability` + the `BBTBRootView.scenePhase=.active` handler + `ServerListViewModel.silentForegroundRefresh`). Many of these fire fire-and-forget on launch from `BBTB_iOSApp.init` and contend for the Mach port ceiling that has already been documented as a crash class in Phase 6c (feedback_nevpn_xpc_mach_port.md).
- **The `MainScreenView` body re-evaluates every second** because `ConnectionTimer` keeps an `autoconnect()`-ed Timer.publish even when the tunnel is disconnected (`since == nil`) â€” the View still receives ticks and re-renders. On the wider main screen that means `StatusPill`, `ConnectionButton`, `ServerLineView`, and the toolbar items get repeatedly diffed by SwiftUI.
- **Three observer hops fan out from a single NEVPNStatus event** (`TunnelController.nevpnObserver`, `MainScreenViewModel.nevpnStatusObserver`, plus the cached-manager refresh from `bbtbProvisionerDidSave`). Each spawns a new Task â€” a moderate but measurable energy drain on `iOS 26` where NEVPNStatusDidChange fires very frequently during reconnects.
- **Resource-heavy SwiftData read patterns**: `countSupportedConfigs` runs `try context.fetch(descriptor).count` (fetches all rows, not a count predicate); `pendingDeleteSubscriptionServerCount` is a computed-property that fetches all rows on every body invocation; `loadFromStore` fetches everything twice (Subscriptions + ServerConfigs) on every server-list refresh. With 50+ servers these copy meaningful work onto the main actor.

These five themes together fully explain the user's "feels heavy since Phase 5" report. Almost every finding below traces back to one of them.
