# Phase 6d Audit — CODEX Pass

## 1. Executive summary

- The main connect tap path is heavy because Auto mode probes every supported server before provisioning and starting the tunnel; this alone can add 500ms+ even before NetworkExtension work begins.
- Cold start does multiple `NETunnelProviderManager.loadAllFromPreferences()` trips and may also copy a trace-level sing-box log synchronously in iOS app init.
- The tunnel extension still has Phase 1 debug behavior in the hot path: trace logging to App Group and repeated JSON parse/validate/expand before libbox start.
- Several background "maintenance" tasks are launched during app init / first active scene and can contend with the initial MainScreen render or first connect.
- Most recommendations preserve Phase 6c invariants: no custom reconnect loops, no XPC in `NEVPNStatusDidChange`, observer queue remains `nil`, and `applyVPNStatus` remains the UI authority.
