# Phase 9 — UI Design Contract (Deep Links)

> Visual and interaction contract for Phase 9 (DEEP-01/02/05). Generated by gsd-ui-researcher; sources — `09-CONTEXT.md` D-01..D-09, `09-RESEARCH.md`, codebase tokens из `BBTB/Packages/DesignSystem/Sources/DesignSystem/DesignSystem.swift`, существующие L10n keys из `BBTB/Packages/Localization/Sources/Localization/L10n.swift`.

**Scope reminder.** Phase 9 — это **routing-фаза**, не UI-фаза. Единственные новые user-visible поверхности:

1. **Error Alert** при невалидной/неудачной deep-link обработке — переиспользует существующий `MainScreenViewModel.lastError` + SwiftUI `.alert` (D-08).
2. **ImportProgressOverlay** во время `ConfigImporter.importFromRawInput(...)` — уже существует (`BBTB/Packages/AppFeatures/Sources/MainScreenFeature/ImportProgressOverlay.swift`), Phase 9 переиспользует без изменений.
3. **Cold-start UX** — нет видимого spinner / splash до того как VM готова (D-09 буферизация через `@State var pendingDeepLink`). Пользователь видит обычный cold-start, затем — спустя ≤1 секунду после `applyInitialStatusSnapshot` — ImportProgressOverlay.

**No new screens. No new icons. No new colors. No new typography.** Контракт сводится к copywriting (новые L10n keys) + reuse существующих DS tokens.

### Voice & tone rules

- **Tone:** объясняющий + actionable. Каждая error copy содержит (1) что произошло простыми словами, (2) что делать дальше (fallback path).
- **No technical jargon в user-facing strings.** В описаниях ошибок — не показывать URL целиком (security: T-09-08 — sensitive data leak via screenshot), не показывать `DeepLinkError` enum case name, не показывать stack traces. Технические детали уходят в `DeepLinksLogger` (Console.app) с `privacy: .private` для URL body.
- **Имя бренда:** «BBTB» / «Верни жука» — НЕ упоминать в alert copy (короче и яснее без бренда).
- **No emojis.** Project-wide правило (CLAUDE.md). Особенно в error messages.

### Empty state

**Not applicable для Phase 9.** Phase 9 не имеет своих экранов с empty state. Если deep-link открывает приложение на пустом MainScreen (нет ни одного сервера в SwiftData), при успешном импорте через DeepLinkRouter Empty state автоматически уходит — это поведение Phase 2 (existing). Если импорт падает — alert показывается поверх Empty state, который остаётся в кадре. Это **корректное и желаемое** поведение (D-08 reuse).

### Destructive actions

**None в Phase 9.** Все user-facing flows — informational (показать ошибку → OK → пользователь сам решает что делать).

### Primary CTA

**Not applicable как «CTA-кнопка».** Единственная user action в Phase 9 поверхностях — Alert OK (dismiss). Это не CTA в маркетинговом смысле; это acknowledgement.

### Cold-start UX copy

**No copy.** D-09 cold-start path **не показывает дополнительного UI** до `applyInitialStatusSnapshot`. Пользователь видит обычный launch screen / window background, потом — спустя ≤1сек после VM-ready — ImportProgressOverlay (с существующим текстом `L10n.importProgress`). Никакого «Loading deep link…» / «Waiting for app to be ready…» не показывать — это создавало бы flicker и confusion.

## Interaction Contract

> Раздел сверх template — необходим для downstream agents (planner / executor / UI-checker / UI-auditor) чтобы понять flow без чтения RESEARCH.md.

### F1 — Cold-start через deep link (D-09)

```
[User taps bbtb://import?url=… in Telegram]
         │
         ▼
[OS launches BBTB app cold]
         │
         ▼
[SwiftUI WindowGroup renders → BBTBRootView appears]
         │  initialManagersApplied == false
         │  → .onOpenURL fires → routeOrBuffer(url) → pendingDeepLink = url (BUFFERED)
         │
         │  [no visible UI change — пользователь видит обычный cold-start launch]
         │
         ▼
[BBTB_iOSApp.init Task chain finishes → applyInitialStatusSnapshot()
 → viewModel.initialManagersApplied = true]
         │
         ▼
[.task block in BBTBRootView fires after wireRulesCoordinator]
         │  if pendingDeepLink != nil:
         │     viewModel.handleDeepLink(pending, router: deepLinkRouter)
         ▼
[MainScreenViewModel.handleDeepLink starts]
         │  importInProgress = true → ImportProgressOverlay shown (existing UI)
         │  ▼
         │  try await deepLinkRouter.handle(url) → ImportHandler → ConfigImporter
         ▼
[Success path: ConfigImporter merges → refresh() → MainScreen updates → overlay dismissed]
   OR
[Failure path: error → lastError set → ImportProgressOverlay dismissed → .alert shown]
```

**Latency budget:** от tap до либо ImportProgressOverlay либо Alert — ≤1.5 sec на iPhone 15-class hardware (cold-start ≤1сек per PERF-01..05 + VM applyInitialStatusSnapshot ≤500ms). На warm start — ≤300ms.

### F2 — Warm-start через deep link

```
[BBTB app уже запущено, пользователь нажимает bbtb://… в другом приложении]
         │
         ▼
[OS sends URL → SwiftUI .onOpenURL fires → routeOrBuffer(url)]
         │  initialManagersApplied == true (since cold-start completed earlier)
         │  → viewModel.handleDeepLink(url, router: …) сразу
         ▼
[Тот же ImportProgressOverlay / Alert flow что и F1]
```

**Latency:** ≤300ms (foreground tap response без cold-start cost).

### F3 — Universal Link (macOS critical path — Pitfall 1)

```
[User taps https://import.bbtb.app/import?url=… in Safari/Mail/Telegram-desktop]
         │
         ▼
[OS routes via NSUserActivity (browse-web type) — NOT .onOpenURL]
         │
         ▼
[SwiftUI .onContinueUserActivity(NSUserActivityTypeBrowsingWeb) fires]
         │  guard activity.webpageURL else return
         │  → routeOrBuffer(url) → same as F1/F2
         ▼
[Идентичный flow ImportProgressOverlay / Alert]
```

**Critical:** Оба модификатора (`.onOpenURL` + `.onContinueUserActivity`) должны быть в **каждом** root view одновременно. UI-checker должен проверить присутствие обоих в `BBTB_iOSApp.swift` и `BBTB_macOSApp.swift`.

### F4 — Error UX (D-08)

```
[Any DeepLinkError из DeepLinkRouter / ImportHandler]
         │
         ▼
[MainScreenViewModel.handleDeepLink catch block]
         │  lastError = error.localizedDescription
         │  importInProgress = false (defer) → ImportProgressOverlay dismissed
         │  ▼
         │  if supportedConfigCount > 0:
         │     state = .error(message: ...)
         │  else:
         │     (Empty state остаётся — alert поверх)
         ▼
[SwiftUI .alert binding triggers — system-rendered]
         │  Title: L10n.alertDeepLinkErrorTitle
         │  Message: lastError (from DeepLinkError.errorDescription)
         │  Action: Button(L10n.actionOK) → lastError = nil
         ▼
[User taps OK → alert dismissed → user остаётся на MainScreen / Empty state]
```

### Reactive states summary

| State | Trigger | Visual |
|-------|---------|--------|
| Idle (no deep link) | Default | Existing MainScreen (no change) |
| Buffered (cold-start pre-VM-ready) | `.onOpenURL` fires before `initialManagersApplied` | No visible change — pendingURL stored in @State |
| Importing | `handleDeepLink` running | `ImportProgressOverlay` (existing, no changes) |
| Imported | Success | Overlay dismissed, MainScreen refreshed (existing behavior) |
| Error | `DeepLinkError` thrown | Alert shown (new copy keys; existing alert binding) |

## Accessibility

Phase 9 наследует accessibility properties от Apple-native компонентов:

- **SwiftUI `.alert`** — VoiceOver-compatible by default; title и message читаются последовательно; OK button auto-traited as `.button`.
- **ImportProgressOverlay** — Phase 2 baseline. `ProgressView` имеет implicit `accessibilityLabel` от system. **Phase 9 проверяет** что VoiceOver читает `L10n.importProgress` при появлении overlay (regression check — не должно сломаться).
- **Dynamic Type** — все DS.Typography tokens declared через `Font.system(.style, design:)` — auto-scaling. Phase 9 не override-ит размеры.
- **Localization** — все user-facing strings через L10n (LOC-01). Ни одного hardcoded English/Russian в новом коде.

**No new accessibility hooks needed** — Apple-native components do the right thing.
