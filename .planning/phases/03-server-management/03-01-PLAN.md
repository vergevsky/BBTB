<objective>
Заложить фундамент multi-subscription: новая SwiftData entity Subscription, расширение ServerConfig полями для ping/identity, idempotent миграция Phase 2 → Phase 3, ConfigImporter создаёт/переиспользует Subscription при импорте subscription URL.

Phase Goal (user story из ROADMAP):

**As a** пользователь BBTB, **I want to** видеть свои серверы сгруппированными по подпискам, переключаться между ними и обновлять список одним жестом, **so that** я мог подключаться к лучшему доступному серверу без ручной настройки.

Этот план — первый вертикальный слайс: end-to-end test «import subscription URL → Subscription row создан + servers с subscriptionID != nil». UI этого слайса (sections) появится в Plan 03; здесь только data foundation, проверяемая через тесты.

Purpose: Без Subscription @Model и FK в ServerConfig невозможны секции в UI (Plan 03), не работает merge-by-identity при re-fetch (Plan 04), не работает cascade delete (Plan 04). Все три зависят от этого фундамента.

- `ServerConfig.swift` — 5 новых optional полей (lightweight migration)
- `SwiftDataContainer.swift` — registration Subscription + idempotent migration

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External subscription URL → App Group SwiftData | Server-controlled string (subscription URL response body, Profile-Title header) пересекает в локальное хранилище — может содержать malicious payload (oversized body, control chars в title, malformed URI) |
| Phase 2 ServerConfig.subscriptionURL → Phase 3 Subscription.url | Existing data становится FK source — миграция должна быть idempotent чтобы повторный запуск не дублировал rows |
