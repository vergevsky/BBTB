## Phase Goal

**As a** user of BBTB, **I want to** open Settings → Advanced and configure my preferred DNS (custom server or AdBlock toggle), **so that** my tunnel uses my chosen DNS strategy with the priority custom > AdBlock > Cloudflare default.

<objective>
Wave 3 (vertical slice) — Settings UI. Two persistent `@AppStorage` settings (`customDNS: String`, `adBlockEnabled: Bool`) and a new `AdvancedSettingsView` accessible via NavigationLink from the existing `SettingsView`. Plus a computed `dnsConfig: DNSConfig` on `SettingsViewModel` that resolves the priority logic from D-01..D-04 — this computed property is the bridge into Wave 5 (TunnelController) and the failover code in Wave 6.

This wave is a **vertical user-facing slice**: after it ships, a user can open the new screen, toggle AdBlock, type a custom DNS, and the value persists. The wired effect on the actual tunnel happens in Wave 5 — but the persistence + priority logic is fully unit-tested here.

Purpose: NET-02 (Custom DNS UI), NET-03 (AdBlock toggle UI), foundation for NET-01 (priority logic resolution).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User text input ↔ SettingsViewModel.customDNS | Untrusted string crosses into model layer. Validation happens here (Task 1 dnsConfig computed) AND at UI layer (Task 2 CustomDNSField error message). |
| @AppStorage UserDefaults ↔ ConfigImporter (Wave 5) | UserDefaults plaintext storage; not sensitive (no secrets). Same key prefix `app.bbtb.` as killSwitchEnabled. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-W3-01 | Tampering / Input Injection | Custom DNS field accepts garbage → flows into sing-box JSON via DNSConfig.custom | mitigate | Task 1: SettingsViewModel.dnsConfig validates IPv4 OR RFC 1123 hostname before constructing .custom; invalid input falls back as if empty (treats as Cloudflare/AdGuard). Task 2: CustomDNSField shows inline error message for invalid input. Two layers of defense (Pitfall 9). |
| T-06-W3-02 | Information Disclosure | UserDefaults stores customDNS in plaintext | accept | DNS preferences are not secrets. App sandbox protects UserDefaults from other apps. Same security posture as killSwitchEnabled (already shipped Phase 2). |
| T-06-W3-03 | DoS | User types extremely long string in customDNS field | mitigate | Hostname validation caps length at 253 chars (RFC 1123 max FQDN length). Invalid → falls back; no buffer issues. |
| T-06-W3-04 | Spoofing | User enters legit-looking malicious DoH host (e.g. attacker's DNS) | accept | Trust model = user-explicit; CustomDNS is a feature, not a security control. Documented in CONTEXT D-03 ("Custom DNS has highest priority"). If user wants to point DNS at attacker's server, that's their choice; we don't validate trustworthiness of arbitrary hostnames. |
</threat_model>
