## Phase Goal

**As a** user of BBTB on iOS/macOS, **I want to** have all IPv6 traffic blackholed inside the tunnel, **so that** my real IPv6 ISP address is never exposed to remote servers when I'm connected to a v4-only VPN exit.

<objective>
Wave 2 (vertical slice) — IPv6 blackhole. Two coordinated changes that together prevent IPv6 leaks at both the OS routing layer (iOS/macOS) and the sing-box engine layer:

1. **`TunnelSettings.makeR6Safe`** — replace `settings.ipv6Settings = nil` (Phase 1 TODO, line 53) with a real `NEIPv6Settings(addresses: ["fd00::1"], networkPrefixLengths: [128])` that captures `::/0` into the TUN. Preserve R6 invariant by NOT setting `destinationAddresses`.

2. **`SingBoxConfigLoader.expandConfigForTunnel`** — extend the runtime-injected TUN inbound's `address` array to include the IPv6 ULA prefix `fd00::1/126` and add `route_address: ["::/0"]` so sing-box's gvisor stack captures v6 packets inside the tunnel where they die (no v6 outbound).

Together, these close the leak documented in 06-RESEARCH.md Pitfall 1: "nil NEIPv6Settings causes IPv6 leak". Without both changes, AAAA-resolvable destinations bypass the tunnel on cellular networks (LTE often has v6; residential Wi-Fi in RU often doesn't).

Purpose: NET-05 (IPv6 tunneled by default = blackholed in Phase 6 per D-06) and NET-06 (block fallback as primary path).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| iOS routing layer ↔ packet flow | NEIPv6Settings.includedRoutes determines which v6 destinations enter our TUN. Misconfig = v6 leak. |
| sing-box gvisor stack ↔ TUN inbound | sing-box's TUN inbound reads v6 packets from the FD; without route_address `::/0` it might bypass the v6 packets to direct. |
