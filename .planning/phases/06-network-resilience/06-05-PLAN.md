## Phase Goal

**As a** user of BBTB, **I want to** have the tunnel auto-reconnect with banner feedback when my network changes or my Mac wakes from sleep, with my chosen DNS strategy applied on every reconnect, **so that** I don't need to manually re-tap Connect every time the network changes.

<objective>
Wave 5 (integration wave) — wire the foundation pieces from Waves 1, 3, 4 into the live flow:

1. **DNSConfig threading**: `ConfigImporter.buildDNSConfig` reads `SettingsViewModel` via `UserDefaults` (since `ConfigImporter` is non-MainActor), derives `DNSConfig` per D-01..D-04, passes it into `PoolBuilder.buildSingBoxJSON(from:dns:)`. End-to-end: user toggles AdBlock → next reconnect uses AdGuard DoH.

2. **TunnelController promotion to actor + observers**: convert `TunnelController` from `final class @unchecked Sendable` to `actor`; add `NetworkReachability`, `ReconnectStateMachine`, `NEVPNStatusDidChange` observer, macOS wake observer (`NSWorkspace.didWakeNotification`), `handleForeground()` for iOS scenePhase, `manualDisconnectInProgress` flag (Pitfall 3 mitigation), `lastSuccessfulConnectAt` (Pitfall 4 mitigation — Wave 6 uses this for failover reset). Preserve existing `connect()`/`disconnect()` contracts.

3. **Banner state extension**: extend `ReconnectBanner` to accept a `message: String` parameter with a backward-compat default (existing kill-switch banner unchanged). `MainScreenViewModel` gains `@Published reconnectBannerState` driven by ReconnectStateMachine observer.

4. **Local notification on .allFailed**: new `UserNotificationsHelper` that requests `.alert + .sound` authorization on first failure (not at launch — UX principle from CLAUDE.md). Posts notification with localized title/body.

5. **scenePhase wiring** in `BBTB_iOSApp.swift`: extend the existing `onChange(of: scenePhase)` block to additionally trigger `tunnelController.handleForeground()`.

Wave 5 is the most context-heavy plan (3 substantive tasks); a sober estimate is ~45% context. It depends on Waves 1, 3, 4 having shipped (Wave 1: PoolBuilder.dns parameter; Wave 3: SettingsViewModel.dnsConfig + UserDefaults keys; Wave 4: NetworkReachability + ReconnectStateMachine actors). Wave 6 (failover) and Wave 5 share TunnelController — Wave 5 lays the foundation and stubs `failoverNext: nil` (so retries-only, no failover yet); Wave 6 replaces the stub with FailoverProvider.

Purpose: NET-01 wiring (DNS user settings → JSON), NET-08/09/10 active triggers.
