## Phase Goal

**As a** user of BBTB, **I want to** have the app automatically recover from network changes (Wi-Fi ↔ LTE, wake from sleep) with bounded retry attempts, **so that** my VPN stays connected through ordinary mobility without me opening the app to manually reconnect.

<objective>
Wave 4 (independent from Waves 1-3 — they touch different files) — Reconnect state machine + reachability. Two new actors in `MainScreenFeature/` that together implement D-07's retry policy:

1. **`NetworkReachability`** — wraps `NWPathMonitor` in an actor. Filters callbacks to physical interfaces (Wi-Fi/Cellular/WiredEthernet) only, throttles to 500ms, dedupes via `lastPhysicalType` tracking. Emits typed events `.satisfied(physical:)`, `.unsatisfied`, `.changed(from:to:)`. Solves Pitfall 2 (monitor firing during our own tunnel bring-up).

2. **`ReconnectStateMachine`** — actor with a single `run(firstAttempt:failoverNext:)` API that drives the retry loop. State exposed via an observer closure (Sendable). Each cycle: 3 attempts with `Task.sleep` of 2/4/8 seconds; on all 3 fails → call `failoverNext` to get the next server's attempt closure (or nil → `.allFailed`). Cancellation via `Task.isCancelled` propagation.

This wave is **autonomous and independent of Waves 1-3** — it can run in parallel with them (we plan it sequential to keep the executor's context window predictable, but `depends_on: []` declares the structural truth). Wave 5 (TunnelController integration) is where these two actors get wired into the existing flow.

Purpose: NET-08 (auto-reconnect on Wi-Fi↔LTE), NET-09 (auto-reconnect on sleep/foreground — partial; the foreground handler is wired in Wave 5), NET-10 (IP change → handled by NWPathMonitor's `.changed` event).

Output:
- `NetworkReachability.swift` (new actor).
- `ReconnectStateMachine.swift` (new actor).
- Two new test files with comprehensive coverage including injected mock clocks where needed.
</objective>
