## Phase Goal

**As a** user of BBTB on iOS/macOS, **I want to** have DNS-over-HTTPS resolution inside the tunnel with a bootstrap chain that avoids Russian DNS providers, **so that** my DNS queries are encrypted, leak-proof, and not subject to ТСПУ enumeration via Yandex/state DNS.

<objective>
Wave 1 (vertical slice) — DNS foundation. Introduce `DNSConfig` value type in `VPNCore`, rewrite `PoolBuilder.dnsBlock()` to accept it, and remove the hardcoded Yandex bootstrap (`tcp://77.88.8.8`) per D-01. The pool builder pipes the dnsConfig into the sing-box JSON `dns.servers` block with bootstrap = server IP (if numeric) or AdGuard fallback (D-01), and tunnel DoH = Cloudflare by default or AdGuard if AdBlock or custom override (D-02..D-04).

This wave delivers a passing unit test suite that proves: (1) bootstrap address respects D-01 priority; (2) tunnel DoH endpoint respects D-02..D-04 priority; (3) no `77.88.8.8` substring anywhere in generated JSON.

Purpose: removes the Yandex hardcode that is a known D-01 violation (Russian state-cooperative DNS in the bootstrap chain), and lays the foundation that Waves 3 (Settings UI) and 5 (TunnelController integration) plug into.

Output:
- `BBTB/Packages/VPNCore/Sources/VPNCore/DNSConfig.swift` (new value type)

</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → SettingsViewModel customDNS field (Wave 3) | Untrusted string input crosses into the sing-box JSON via DNSConfig.custom(...). Validation happens in Wave 3 / Wave 5; Wave 1 only carries the value as opaque String. |
| sing-box dns-bootstrap → upstream DNS resolver | Pre-tunnel DNS visible to ТСПУ; D-01 mandates server-IP-first to minimize observable DNS queries. |
| sing-box dns-remote (DoH) → cloudflare-dns.com / adguard-dns.com | Inside tunnel, TLS-protected. sing-box validates DoH server certs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-W1-01 | Tampering | DNSConfig.custom(address:) passthrough in dohAddress() | mitigate | Wave 1 carries the string verbatim; Wave 3 (Settings UI) and Wave 5 (ConfigImporter.buildDNSConfig) validate IPv4-or-RFC1123-hostname before constructing `.custom`. Wave 1 documents this contract in DNSConfig header doc. |
| T-06-W1-02 | Information Disclosure | dns-bootstrap address visible to ТСПУ pre-tunnel | mitigate | D-01: prefer server IP (zero DNS lookup); AdGuard as fallback (international DNS, not state-cooperative). Yandex hardcode (T-06-W1-02 root cause) removed by Task 2. |
| T-06-W1-03 | Spoofing | Cloudflare/AdGuard MITM via cert swap | accept | sing-box delegates TLS cert validation to Go's `crypto/tls`; trust roots are platform default. R1 invariant: no `clash_api` or HTTP server in our JSON, so no parallel attack surface. |
| T-06-W1-04 | Repudiation | DNSConfig changes not logged | accept | DNS settings are user preference, not security event. Wave 3 may add OSLog noting "DNS strategy changed"; not required for MVP. |
</threat_model>
