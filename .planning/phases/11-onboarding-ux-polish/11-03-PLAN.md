    3. **Important sequencing:** новый `.fullScreenCover(Onboarding)` modifier ДОЛЖЕН быть в chain ДО `.fullScreenCover($showQRScanner)`. SwiftUI не разрешает два `.fullScreenCover` на одной view — если оба активны, top-most в chain побеждает. Гарантируем: после `hasShownOnboarding = true` setter (вызван из onDismiss), `!hasShownOnboarding` становится `false`, Onboarding sheet закрывается → теперь QR sheet может открыться независимо. Параллельно открыть QR из Onboarding (через `onScanQR` closure) — нюанс: closure setает `showQRScanner = true`, но onboarding всё ещё active. Чтобы QR scanner появился, нужно сначала закрыть onboarding. Решение: внутри `onScanQR` closure СНАЧАЛА выставить `hasShownOnboarding = true` (D-03 спецификация говорит «после import» — но QR scanner это шаг к import; user уже committed). Уточнение: оставить как `{ showQRScanner = true }` без preset hasShownOnboarding — после успешного QR-импорта `viewModel.state` обновится → `dismissIfImported` в OnboardingView сработает → `onDismiss` сработает → `hasShownOnboarding = true` → onboarding sheet закроется → QR sheet остаётся открытым только если он не успел закрыться сам. SwiftUI поддерживает «sheet over sheet» только если они на разных view layers. Альтернатива: `onScanQR = { hasShownOnboarding = true; showQRScanner = true }` — это гарантирует, что QR scanner откроется независимо от import success. Реализовать ИМЕННО ТАК (compromise, чтобы избежать sheet-over-sheet race).

       Финальная версия onScanQR closure:
       ```
       onScanQR: {
           // Закрыть onboarding до открытия QR (избежать sheet-over-sheet)
           hasShownOnboarding = true
           showQRScanner = true
       }
       ```
